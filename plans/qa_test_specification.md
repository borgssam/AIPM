# 기능 및 품질 검수 테스트 명세서 (QA & Test Specification)

본 문서는 **AI PM Phase 1** 시스템의 완성도와 사양 충족 여부를 확인하기 위해 수립된 품질 검수 및 인수 테스트 명세서입니다. 본 설계에 기재된 테스트 시나리오를 충족해야 최종 제품 배포가 가능합니다.

---

## 1. 자동화 테스트 계획 (Automated Unit & Integration Tests)

테스트 프레임워크로는 파이썬 `pytest`와 `httpx` 비동기 테스트 클라이언트를 사용합니다.

### 1.1 주요 단위 테스트 케이스 (Unit Test Cases)
1.  **인증 테스트 (`tests/test_auth.py`):**
    *   `test_user_signup_success`: 정상 정보 입력 시 회원가입 완료 및 비밀번호 해싱 검증.
    *   `test_user_signup_duplicate`: 이미 존재하는 username으로 가입 시도 시 `400 Bad Request` 예외 발생 검증.
    *   `test_user_login_success`: 올바른 자격증명 제출 시 유효 기간이 설정된 JWT 액세스 토큰 발급 검증.
    *   `test_user_login_fail`: 잘못된 비밀번호 입력 시 `400 Bad Request` 에러 검증.
    *   `test_jwt_expiration`: 만료 시간이 지난 토큰으로 API 호출 시 `401 Unauthorized` 예외가 정상 감지되는지 검증.

2.  **스케줄 자동 생성 테스트 (`tests/test_schedule.py`):**
    *   `test_schedule_generate_success`: 마크다운 명세서 파일 내용을 전달했을 때 AI PM이 태스크를 정상적으로 분해하여 `[시작일/마감일/내용/우선순위]`가 채워진 칸반 티켓 및 매핑된 `qa_inspection_items`를 DB에 벌크 삽입하는지 검증.
    *   `test_schedule_generate_conflict_detection`: 기획서와 기존 API_SPECIFICATION.md 간의 논리적 충돌이 있을 때, `[AI-Detected]` 태그를 가진 경고 티켓이 `TO_REVIEW` 열로 등록되고 슬랙 알림 트리거가 1회 작동하는지 검증.
    *   `test_schedule_generate_parse_error`: 유효하지 않거나 깨진 마크다운 전달 시 `400 Bad Request` 에러 상세 응답 반환 검증.

3.  **티켓 관리 및 권한 제어 테스트 (`tests/test_tickets.py`):**
    *   `test_ticket_update_general`: 티켓 정보 PATCH API(`PATCH /api/v1/tickets/{ticket_id}`)를 통한 일정, 내용, 우선순위 변경 정상 작동 검증.
    *   `test_rbac_assignee_change`: `assignee_id`를 수정할 때, 요청 세션의 사용자가 PM 권한이 아닐 경우 `403 Forbidden`이 발생하는지 검증.
    *   `test_rbac_status_change_forbidden`: 배정되지 않은 타인의 티켓 상태를 일반 팀원 권한으로 강제 수정 시도 시 `403 Forbidden` 발생 검증.

4.  **품질 검수 하드 게이트 테스트 (`tests/test_hard_gate.py`):**
    *   `test_resolve_ticket_success`: 티켓에 연결된 모든 `qa_inspection_items` 상태가 `PASS`인 경우, `POST /api/v1/tickets/{ticket_id}/resolve` 호출 시 상태가 정상적으로 `DONE`으로 변경되고 해결 방안이 컨텍스트에 동기화되는지 검증.
    *   `test_resolve_ticket_hard_gate_fail`: 링크된 검수 항목 중 하나라도 `PASS`가 아니거나 `UNTESTED` 상태일 때, `/resolve` API 또는 티켓 상태 `DONE` 변경 시도 시 백엔드가 즉시 `400 Bad Request` (메시지: `"모든 연계 검수 체크리스트 항목이 완료(PASS) 상태여야 합니다."`) 오류를 반환하며 작업을 차단하는지 검증.

## 2. 수동 인수 테스트 시나리오 (Manual Acceptance Test Cases)

### 2.1 [TC-1] 명세 파일 업로드 및 일정 일괄 생성 검증
*   **사전 조건:** 브라우저에서 PM 권한(`role='PM'`)을 가진 계정으로 로그인한 상태.
*   **수행 단계:**
    1.  상단 명세서 파일 업로드 영역에 `요구명세서.md` 및 `기능명세서.md` 파일을 각각 드래그앤드롭하여 추가한다.
    2.  `[개발 스케줄 생성 및 상충 분석]` 버튼을 클릭한다.
*   **기대 결과:**
    -   스피너 로딩 바가 나타난 후, 분석이 종료되면 칸반보드 `To Do` 열에 분석 내용에 기초한 개발 티켓(시작일, 마감일, P1 우선순위 배지 노출)이 일괄 렌더링되어야 한다.

### 2.2 [TC-2] AI 상충 감지 및 슬랙 알림 검증
*   **사전 조건:** `API_SPECIFICATION.md`의 기존 스펙과 모순되는 새로운 기획 문서를 준비한다.
*   **수행 단계:**
    1.  상충 요인이 포함된 명세서를 파일 업로드 영역에 투입하고 분석을 실행한다.
*   **기대 결과:**
    -   칸반보드의 `To Review (검토 필요)` 열에 🤖 아이콘과 굵은 테두리가 강조된 `[AI-Detected] API 스펙 충돌 경고` 카드가 생성되어야 한다.
    -   지정된 Slack 메신저 채널에 *"기획 충돌 이슈가 등록되었습니다. 칸반보드를 확인해 주세요."*라는 알림 브리핑 메시지가 실시간으로 수신되어야 한다.

### 2.3 [TC-3] 칸반보드 필터링 및 검색 인터랙션 검증
*   **사전 조건:** 보드에 다수의 개발 카드와 경고 카드가 렌더링되어 있는 상태.
*   **수행 단계:**
    1.  검색바에 특정 키워드(예: "로그인")를 입력해 본다.
    2.  `내 작업만 보기` 체크박스를 토글해 본다.
    3.  우선순위 필터를 `P0`으로 변경해 본다.
*   **기대 결과:**
    -   입력된 키워드나 우선순위, 사용자 할당 필터에 부합하는 카드만 보드 상에 남고 즉시 필터링 렌더링되어야 한다.

### 2.4 [TC-4] 담당자 배정 권한 및 드래그앤드롭 제약 검증
*   **사전 조건:** 일반 개발자 계정(`role='DEVELOPER'`)으로 신규 로그인하여 보드에 진입한다.
*   **수행 단계:**
    1.  특정 카드를 클릭하여 드롭다운 메뉴를 연다.
    2.  담당자를 PM 권한으로 강제 변경하려 하거나, 담당자가 지정되지 않은 다른 팀원의 카드를 드래그하여 `In Progress`로 옮겨 본다.
*   **기대 결과:**
    -   일반 개발자 권한으로는 담당자 변경 드롭다운 변경 저장 시 에러 팝업("권한이 없습니다")이 발생해야 한다.
    -   타인 소유의 카드 드래그 앤 드롭 상태 변경 시 화면이 원래 컬럼으로 원복되며 에러 안내가 발생해야 한다.

### 2.5 [TC-5] 상세 모달창 품질 검수 하드 게이트 검증
*   **사전 조건:** 김개발(`DEVELOPER`)로 로그인하여 나에게 할당된 `In Progress` 상태 카드의 모달창을 오픈한다.
*   **수행 단계:**
    1.  상세 모달 하단의 품질 검수 체크리스트(4개) 중 3개만 체크한다.
    2.  해결 방안(TextArea)에 조율 완료 내역을 입력한다.
    3.  하단의 `[완료 (Done)]` 버튼 상태를 확인한다.
    4.  4번째 체크박스까지 마저 체크한 뒤 완료 버튼을 클릭한다.
*   **기대 결과:**
    -   체크박스가 1개라도 해제되어 있는 동안에는 `[완료]` 버튼이 연회색으로 비활성화(`disabled`)되어 있으며 마우스 포인터가 금지 상태여야 한다.
    -   4개의 항목이 모두 체크되는 즉시 버튼이 녹색으로 활성화되며 클릭 가능해지고, 완료 클릭 시 API를 호출하여 카드가 `Done` 열로 정상 이동해야 한다.

### 2.6 [TC-6] 전역 프로젝트 설정 및 웹훅 갱신 검증
*   **사전 조건:** `PM` 계정으로 로그인하여 `설정` 탭으로 이동한다.
*   **수행 단계:**
    1.  슬랙 웹훅 URL 입력란의 텍스트를 다른 슬랙 채널 웹훅으로 수정한 뒤 `[설정 저장]` 버튼을 누른다.
    2.  다시 기획서 분석을 통해 상충 이슈를 유발시킨다.
*   **기대 결과:**
    -   설정 변경 후 저장 완료 팝업이 노출되어야 한다.
    -   상충 발생 시, 변경한 신규 슬랙 채널 채널로 정상적으로 긴급 알림 메시지가 발송되어야 한다.

### 2.7 [TC-7] 세션 만료 로그아웃 검증
*   **수행 단계:**
    1.  브라우저 우측 상단의 `[로그아웃]` 버튼을 클릭한다.
*   **기대 결과:**
    -   로컬 스토리지의 사용자 토큰이 즉시 삭제되고 메인 대시보드 접근이 제한되며 `/login` 화면으로 리다이렉트되어야 한다.
